SOLID, um porque aplicar este conceito nos projetos.

 Padronização do código:
 - Separar as camadas do projeto, cada camada com o seu propósito, uma camada para conexão
com o banco, uma para insersão do cliente, uma para validação do cliente.

 - Padronizar a nacionalidade de métodos, variáveis e classes. Se for proposto o projeto em
português, implementar tudo com nomes em português. Como exemplo, nunca chamar um método de 
"GetByDia();".

 - Padronizar as instâncias de classes e chamadas de Métodos. Sempre que instanciar uma classe, deve-se usar tudo que pertence aquela classe e se aplica em métodos também.

 - Usando essa padronização torna-se muito mais fácil de se testar um código, fazer alterações, fornece muito mais o aproveitamento e torna seu código possível para permanecer o máximo de tempo sendo utilizado.

Sobre DDD.

 - DDD são boas práticas no desenvolvimento do código de um projeto.
 - Para implementar o DDD temos que entender o negócio.
 - Delimitar os contextos da modelagem estratégica do projeto.
 - Proporcionar mais facilidades para entender tudo o que tiver de complexo no código.
 - Dentro do projeto existe o Time de Desenvolvimento e o Domain Experts,são esses que entendem sobre o negócio e vão guiar o time para que desenvolva um bom projeto de acordo com os processos certos e para que não tenha duvidas.
